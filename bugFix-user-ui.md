# 📝 สรุปความคืบหน้าและการเปลี่ยนแปลงใน Branch: `bugfix/user-ui`

เอกสารฉบับนี้จัดทำขึ้นเพื่อแสดงรายละเอียดการพัฒนา การแก้ไขบั๊ก และการเพิ่มฟีเจอร์ต่าง ๆ ภายใน Branch `bugfix/user-ui` ซึ่งมุ่งเน้นไปที่การยกระดับความปลอดภัย (Security), เสถียรภาพในการทำงานแบบ Concurrent (Concurrency & Thread-Safety), ประสบการณ์ของผู้ใช้งาน (UI/UX) และการปรับปรุงเอกสารโครงสร้างระบบทั้งหมดให้สมบูรณ์พร้อมใช้งาน

## 🚀 ภาพรวมการเปลี่ยนแปลง (High-Level Summary)

การดำเนินงานใน Branch นี้ครอบคลุม 4 เสาหลักสำคัญของโครงการ:
1. **Security & Data Integrity**: ป้องกันช่องโหว่ Ghost User, การจำกัดความล้มเหลวของการยืนยัน OTP (Brute Force Defense) และการควบคุม Token ผ่าน Authorization header
2. **Concurrency & Double-Spend Prevention**: ใช้ระบบ Pessimistic Locking (`FOR UPDATE`) ในฐานข้อมูลเพื่อป้องกันสิทธิ์แต้มติดลบจากการกดแลกคูปองพร้อมกัน และล็อกสถานะคูปองเพื่อไม่ให้เกิดการสแกนซ้ำที่ตู้จำหน่ายสินค้า (Double-Redemption)
3. **Physical Machine Integration**: ปรับรูปแบบคูปองให้เป็นระบบตัวเลข 8 หลัก (Strict 8-Digit Numeric) เพื่อให้ผู้ใช้นำไปกดใช้งานผ่านตู้กดหน้าร้านที่มีเพียงปุ่มตัวเลขได้อย่างสะดวก
4. **PWA & Offline Support**: เพิ่ม Service Worker และ PWA Manifest ให้กับ Web UI (`web/web-ui`) เพื่อรองรับการทำงานแบบออฟไลน์ โหลดหน้าเว็บได้เร็วขึ้น และสามารถติดตั้งเป็นแอปพลิเคชันบนมือถือได้
5. **API Design & Complete Documentation**: อัปเกรดไฟล์โครงสร้าง Swagger YAML เพื่ออธิบายชุด API สำหรับ Admin และสิทธิ์ Superadmin ทั้งระบบ พร้อมปรับแก้เอกสารการติดตั้งระดับ Enterprise ใน `README2.md` และ `how-to-run.md`

## 🔍 รายละเอียดการเปลี่ยนแปลงเชิงลึก (Detailed Changes)

### 1. ส่วนของ Backend & Database (`server/app/` & `database/`)

#### 🛡️ ระบบป้องกันสิทธิ์แลกซ้ำและความปลอดภัยของคะแนนสะสม (Anti-Double-Spend & Thread Safety)
* **Pessimistic Locking ในการแลกคูปอง (`server/app/api/orders.py`)**:
  เพิ่ม `FOR UPDATE` ในการคิวรีเช็กคะแนนผู้ใช้และข้อมูลโปรโมชันในฐานข้อมูล เพื่อป้องกันสภาวะแข่งขัน (Race Conditions) เมื่อผู้ใช้งานส่งคำร้องแลกคะแนนสะสมเป็นคูปองเข้ามาพร้อม ๆ กันหลายครั้ง
  ```sql
  -- ป้องกันคะแนนติดลบจากการสปริงแลกคูปองขนาน
  SELECT user_id, points FROM users WHERE phone_number = %s FOR UPDATE;
  SELECT promotion_id, code, points_cost, is_active, expire_date, max_uses FROM promotions WHERE promotion_id = %s FOR UPDATE;
  ```
* **คำนวณแต้มแบบ Atomic (`server/app/api/members.py`)**:
  เปลี่ยนวิธีเพิ่มแต้มให้สมาชิกจากเดิมที่นำแต้มเก่ามาคำนวณบน Python แล้วบันทึกกลับ เป็นการเพิ่มแต้มในคำสั่ง SQL โดยตรงเพื่อป้องกันแต้มทับซ้อนกันในสภาวะ Concurrent:
  ```sql
  UPDATE users SET points = points + %s, last_use = NOW() WHERE user_id = %s;
  ```
* **ระบบป้องกัน Double-Spend บนเครื่องตู้จำหน่ายสินค้า (`server/app/services/coupon_service.py`)**:
  เพิ่มตัวเช็กสถานะการใช้งานคูปอง โดยหากมีออเดอร์ในสถานะกำลังชำระเงิน (`pending_payment`) ที่ผูกกับคูปองใบนั้นอยู่ ระบบจะสกัดกั้นการเรียกใช้งานซ้ำและตอบกลับด้วยสถานะ `"in_use"` เพื่อไม่ให้ออเดอร์อื่นใช้คูปองเดียวกันได้
  * แสดงข้อความภาษาไทยเตือนผู้ใช้: `"คูปองนี้กำลังถูกใช้งานในรายการสั่งซื้ออื่น กรุณารอสักครู่หรือยกเลิกรายการเดิม"`

#### 🔢 อัปเกรดระบบคูปองรองรับแป้นกดตัวเลขของตู้จำหน่ายสินค้า (Numeric Keypad Coupons)
* **ปรับเปลี่ยนโครงสร้างรหัสคูปอง (`server/app/api/orders.py`)**:
  เปลี่ยนการสุ่มรหัสคูปองทางกายภาพจากสัญลักษณ์ตัวอักษรผสมตัวเลข (เช่น `CP-AB12CD34`) มาใช้ **ตัวเลขล้วน 8 หลัก** (`string.digits`) เพื่อความเข้ากันได้ 100% กับระบบแป้นพิมพ์ตัวเลข (Keypad) บนตู้จำหน่ายสินค้า

#### 🔒 ป้องกันการเดา OTP (Brute-Force Protection)
* **การบันทึกจำนวนครั้งที่กรอกผิด (`database/init.sql` & `server/app/services/otp_service.py`)**:
  * เพิ่มฟิลด์ `failed_attempts` ในตาราง `otp_sessions`
  * เมื่อกรอกรหัส OTP ผิด จะทำการอัปเดตจำนวนครั้งสะสมในฐานข้อมูล
  * หากมีการกรอกผิดครบ **5 ครั้ง** ระบบจะบล็อกการใช้งาน OTP ชุดนั้นทันทีและแสดงข้อความแจ้งเตือน: `"รหัส OTP นี้ถูกล็อกเนื่องจากใส่รหัสผิดเกินกำหนด กรุณาขอรหัสใหม่"`

### 2. ส่วนของ Frontend (`web/web-ui/`)

#### 🛑 การป้องกันช่องโหว่ Ghost User (Secure User Onboarding Flow)
* **อัปเดตลำดับการจัดการเซสชัน (`web/web-ui/app/(auth)/login/page.tsx`)**:
  * **ปัญหาเดิม**: มีการบันทึกโทเค็นเข้าสู่คุกกี้ (`saveSession`) ก่อนที่จะมีการบันทึกบัญชีสมาชิกเข้าสู่ระบบ ส่งผลให้เกิดบัญชีผี (Ghost User) ที่ค้างอยู่ในหน้าเว็บแต่ไม่มีโปรไฟล์จริงในฐานข้อมูลหากปิดหน้าต่างสมัครสมาชิกระหว่างทาง
  * **การแก้ไข**: ย้ายส่วนการทำงานให้บันทึกเซสชันลงคุกกี้เฉพาะกรณีที่ยืนยันว่าโปรไฟล์สมาชิกมีอยู่จริงในระบบ (`member.found`) หรือภายหลังจากส่งคำขอบันทึกข้อมูลลงทะเบียนสำเร็จแล้วเท่านั้น
* **ส่งโทเค็นยืนยันตัวตนในหัวคำขอสมัครสมาชิก (`web/web-ui/lib/api/members.ts`)**:
  ปรับโครงสร้างฟังก์ชัน `registerMember` ให้สามารถรับ `token` เป็นอาร์กิวเมนต์ เพื่อใส่ใน `Authorization: Bearer <token>` ทำให้ฝั่งเซิร์ฟเวอร์สามารถตรวจสอบสิทธิ์ก่อนอนุญาตให้สมัครสมาชิกเสร็จสมบูรณ์

#### 🎨 ปรับปรุงประสิทธิภาพของส่วนประสานผู้ใช้งาน (UI/UX Optimizations)
* **การล้างข้อมูลเบอร์โทรศัพท์ (Phone Number Normalization)**:
  อัปเกรดระบบให้สามารถดักจับเบอร์ที่มีคำนำหน้าด้วยรหัสประเทศ `+66` หรือ `66` เพื่อแปลงกลับเป็นตัวเลขศูนย์หลักแรกสุดของประเทศให้โดยอัตโนมัติ ช่วยลดปัญหาความผิดพลาดตอนสแกนหรือกรอกข้อมูลผิดฟอร์แมต
* **ลดการเรนเดอร์ React ที่ไม่จำเป็น**:
  แก้ไข Dependency ในหน้าล็อกอินเพื่อให้ระบบจับเวลา Resend OTP ทำงานเฉพาะในสภาวะที่มีตัวเลขเวลาเหลืออยู่เท่านั้น (`resendSeconds > 0`) แทนการกระตุ้นดีเพนเดนซีทุกวินาที
* **จำกัดความยาวของข้อมูล**:
  เพิ่มแอตทริบิวต์ `maxLength={100}` ในอินพุตชื่อแสดงผล ป้องกันความผิดพลาดทางโครงสร้าง UI ในอนาคต
* **การโฟกัสช่องกรอก OTP (`web/web-ui/components/Ui/OtpInput.tsx`)**:
  ปรับให้เคอร์เซอร์กระโดดไปช่องถัดไปก็ต่อเมื่อมีข้อมูลตัวเลขถูกพิมพ์เข้ามาจริง ๆ เท่านั้น เพื่อขจัดปัญหาช่องถัดไปเด้งโฟกัสก่อนเวลาอันควร

#### 📱 PWA & Service Worker Integration (Offline Support)
* **การลงทะเบียน Service Worker (`public/sw.js` & `ServiceWorkerRegister.tsx`)**:
  เพิ่มความสามารถในการทำงานแบบออฟไลน์ด้วย Caching Strategies เช่น `Network-First` สำหรับ API และหน้า HTML และ `Stale-While-Revalidate` สำหรับไฟล์ Static Assets (ภาพ ฟอนต์ CSS)
* **PWA Manifest (`public/manifest.json`)**:
  เพิ่มไฟล์ Manifest พร้อมกับ Metadata และ `themeColor` ใน `app/layout.tsx` เพื่ออนุญาตให้ผู้ใช้บันทึกแอปเป็นไอคอนบนหน้าจอโฮม (Add to Home Screen) ทำให้ Web UI ทำงานได้เหมือนแอปพลิเคชันมือถือ (Native App Experience)

### 3. ส่วนของ Swagger API & ระบบเอกสาร (`swagger.yaml` & Documentation)

#### 🛡️ ออกแบบระบบสิทธิ์ Admin Authentication
* **เพิ่มโครงสร้าง API สำหรับผู้ดูแลระบบ (Admin) ลงใน `swagger.yaml`**:
  * `/api/admin/auth/login`: ตรวจสอบสิทธิ์ผู้ดูแลระบบด้วยอีเมลและรหัสผ่านเพื่อรับ JWT โทเค็น
  * `/api/admin/auth/register`: สำหรับผู้ดูแลระบบรายใหม่กรอกข้อมูลลงทะเบียนด้วย Token คำเชิญ
  * `/api/admin/auth/invite` (เฉพาะ Superadmin เท่านั้น): ออกลิงก์เชิญชวนผู้ดูแลระบบท่านอื่นเข้ามาใช้งานร่วม
  * `/api/admin/auth/revoke/{admin_id}` (เฉพาะ Superadmin เท่านั้น): ระงับและลบบัญชีผู้ดูแลระบบรายอื่นออก
  * `/api/admin/auth/me`: ดึงข้อมูลประวัติผู้ใช้ของผู้ดูแลระบบปัจจุบัน
  * `/api/admin/auth/admins` (เฉพาะ Superadmin เท่านั้น): เรียกดูรายการบัญชีผู้ดูแลระบบทั้งหมด
* **ปรับปรุง Endpoint สมาชิกใน Swagger**:
  * เพิ่ม API `/api/members/{phone}/coupons/{user_promo_id}/reveal` เพื่อสร้าง/เปิดเผยรหัสตัวเลข 8 หลักสำหรับการนำไปสแกนที่ตู้กดสินค้า
  * ปรับเปลี่ยนรูปแบบการอ้างอิง JWT Security ทั่วทั้งไฟล์โดยระบุให้สอดรับกับการยืนยันตัวตนแบบ Bearer Token ทั่วไป
