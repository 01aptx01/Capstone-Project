# วิธีรันโปรเจกต์ Smart Vending (ฉบับอัปเดตล่าสุด)

คู่มือนี้เป็นจุดอ้างอิงหลักสำหรับการสั่งทำงาน (Run) และพัฒนาโปรเจกต์ **Smart Vending Machine** ในทุกระดับการทำงาน ตั้งแต่การทดสอบด้วย Docker เต็มสแต็ก, พัฒนาแยกทีละบริการ (Hot-Reload), การเชื่อมฮาร์ดแวร์จริงบน LAN/Raspberry Pi, การทดสอบชำระเงิน ตลอดจนการคลี่คลายปัญหาทั่วไป

ภาพรวมสถาปัตยกรรมและรายละเอียดโมดูลเชิงลึกสามารถอ่านได้ที่: [README2.md](README2.md)

## 1. ภาพรวมบริการทั้งหมดในระบบ

ระบบนี้ถูกรวมศูนย์และตั้งค่าผ่าน [docker-compose.yml](docker-compose.yml) ให้พร้อมใช้งานได้ในสัญญาสั่งเดียว ประกอบไปด้วยบริการทั้งหมด 7 ตัว ดังนี้:

| Service | Container Name | หน้าที่การทำงาน |
| :--- | :--- | :--- |
| `db` | `vending-db` | MySQL 8 — สร้าง Schema และใส่ Seed ข้อมูลอัตโนมัติจาก [database/init.sql](database/init.sql) เมื่อ volume ยังว่างเปล่า |
| `server` | `vending-server` | Flask Central Backend API + Socket.IO Gateway + Flasgger API Docs |
| `client` | `vending-pi` | Telemetry & Hardware Agent (รันโค้ด Python จำลองฮาร์ดแวร์ GPIO/NFC บน Docker) |
| `machine-ui` | `vending-machine-ui` | Next.js แอปพลิเคชันหน้าตู้จำหน่ายสินค้า (Kiosk UI) สำหรับลูกค้าสแกน/กดซื้อ |
| `admin-ui` | `vending-admin-ui` | Next.js ระบบจัดการหลังบ้านของแอดมิน (CRUD สินค้า, ตู้, และสต็อกช่อง) |
| `web-ui` | `vending-web-ui` | Next.js แอปพลิเคชันบนมือถือของลูกค้าสมาชิกแบบ PWA (Offline Support, OTP Login, ดูคะแนนสะสม, แลกคูปอง) |
| `swagger-ui` | `vending-swagger-ui` | Swagger UI เพื่อเปิดศึกษาโครงสร้าง API จากเอกสารสัญญากลาง [swagger.yaml](swagger.yaml) |
| `compose-banner` | `vending-compose-banner` | บริการพิมพ์แบนเนอร์แสดงตารางลิงก์พอร์ตต่างๆ ในหน้าต่าง console |

## 3. ตั้งค่า Environment (.env) ครั้งแรก

### 3.1 สร้างไฟล์ `.env` ที่โฟลเดอร์หลัก (Root)
ไฟล์นี้จะอยู่ระดับเดียวกันกับ [docker-compose.yml](docker-compose.yml) โดย Compose จะดึงค่าไปป้อนให้แก่ Next.js ในขั้นตอนการ Build (Build-time Args) และจะป้อนให้เซิร์ฟเวอร์ Flask ในขณะเริ่มทำงาน (Runtime Env)

* **ระบบ Windows (PowerShell):**
  ```powershell
  copy .env.example .env
  ```
* **ระบบ macOS / Linux:**
  ```bash
  cp .env.example .env
  ```

> [!WARNING]
> ไม่ต้องพยายามไปแก้ไขหรือใช้ไฟล์ `client/agent/.env` สำหรับโหมด Docker compose หลัก เนื่องจากตัวแปรสำหรับควบคุม Agent ทั้งหมดจะถูกดึงไปจากไฟล์ `.env` ที่ root นี้โดยตรง

### 3.2 ค่าคอนฟิกเริ่มต้นที่จำเป็นต้องป้อนทันที

เปิดไฟล์ `.env` และกรอกค่า Omise Test Keys ของคุณ (หรือใช้คีย์ทดสอบใดๆ หากต้องการข้ามและจำลองด้วย mock payment):
```env
NEXT_PUBLIC_OMISE_PUBLIC_KEY=pkey_test_5...
OMISE_SECRET_KEY=skey_test_5...
```
*หากต้องการข้ามการส่ง SMS สมาชิกเข้าเบอร์โทรจริงขณะพัฒนา ให้ตั้งค่า `AUTH_DEV_BYPASS=1` หรือปล่อยว่างไว้เพื่อรับรหัส OTP สดทางหน้า Terminal*

## 4. สถานการณ์ A — Docker Compose เต็มสแต็ก

เหมาะสำหรับการรันระบบทั้งหมดขึ้นมาร่วมกันเพื่อทดสอบระบบและตรวจสอบการไหลของข้อมูลตั้งแต่ต้นจนจบ (End-to-End)

```bash
# คัดลอกและตั้งค่า Env ล่วงหน้าตามหัวข้อที่ 3 จากนั้นสั่งรันพร้อมบิลด์
docker compose up --build
```
ระบบจะทะยอยเริ่มทำงาน โดยบริการ `db` และ `server` จะต้องรอให้กระบวนการตรวจสอบความพร้อมของการเชื่อมต่อ (Healthcheck) ผ่านด่านก่อน จากนั้นหน้าจอหน้าตู้ (`machine-ui`) และแอดมินบอร์ด (`admin-ui`) จึงจะเสร็จสิ้นการตั้งค่า

**หากต้องการสั่งให้รันทำงานเป็นงานเบื้องหลัง (Background/Detached Mode):**
```bash
docker compose up --build -d
```

**หากต้องการเฝ้าติดตามบันทึกการส่งเหตุการณ์ของกล่องฮาร์ดแวร์จำลอง (Agent Logs):**
```bash
docker logs -f vending-pi
```

## 5. ตาราง URL และพอร์ตที่เผยแพร่บน Host

เมื่อรันผ่าน Docker Compose บนเครื่องโฮสต์ของคุณ คุณสามารถเรียกใช้งานแอปพลิเคชันต่างๆ ได้ที่ที่อยู่ต่อไปนี้:

| ส่วนประกอบ / บริการ | ที่อยู่ URL บนบราวเซอร์โฮสต์ | รายละเอียดเพิ่มเติม |
| :--- | :--- | :--- |
| **Machine UI** (ลูกค้าหน้าตู้) | [http://localhost:3000](http://localhost:3000) | หน้าจอลูกค้าหน้าร้าน (Kiosk Mode) |
| **Admin UI** (แดชบอร์ดจัดการ) | [http://localhost:3001](http://localhost:3001) | ระบบจัดการหลังบ้านของแอดมิน |
| **Web UI** (แอปสะสมคะแนนสมาชิก) | [http://localhost:3002](http://localhost:3002) | หน้าจอมือถือสมาชิก (ล็อกอินเบอร์โทร) |
| **Flask Server API** | [http://localhost:8000](http://localhost:8000) | REST API และ Gateway Socket.IO |
| **Flasgger Documentation** | [http://localhost:8000/apidocs](http://localhost:8000/apidocs) | สัญญา API สดที่สร้างจากตัวเซิร์ฟเวอร์ |
| **Swagger UI** (OpenAPI 2.0 Spec) | [http://localhost:8081](http://localhost:8081) | แสดงรายละเอียด API จากไฟล์สัญญากลาง |
| **Hardware Agent** (Pi Agent) | [http://localhost:5000](http://localhost:5000) | สำหรับเช็คความเรียบร้อยระดับ Agent |
| **MySQL Database** | `127.0.0.1:3307` | พอร์ตเชื่อมต่อสำหรับ Database GUI (เช่น DBeaver) |

## 6. รายการตรวจสอบความสมบูรณ์ของระบบ (Purchase Flow Checklist)

หากระบบสตาร์ทขึ้นมาสมบูรณ์ โฟลว์การซื้อสินค้าแบบไร้รอยต่อควรเป็นไปตามรายการตรวจสอบนี้:

1. **การยืนยัน WebSocket ของ Agent**: เปิดดู logs ของ `client` (`vending-pi`) จะต้องขึ้นสถานะ `✅ [WS] connected to server` และไม่มีบรรทัดพยายามต่อเชื่อมล้มเหลวซ้ำๆ (Authentication retry loop)
2. **หน้าจอหลัก Machine UI**: เปิด [http://localhost:3000](http://localhost:3000) บนบราวเซอร์ จะต้องไม่ปรากฏหน้าจอข้อความระบบขัดข้องเตือนขึ้นมา และต้องสามารถเลือกหยิบของใส่ตะกร้าได้
3. **การชำระเงินและจ่ายสินค้า**: เมื่อทำรายการชำระเงินเรียบร้อยแล้ว (ไม่ว่าจะสแกน PromptPay จริงหรือคลิก mock-pay บน Terminal) กล่อง Agent จะต้องมีข้อมูลตอบรับ `job.start` และหน้าจอ Next.js หน้าร้านจะต้องเปลี่ยนสเตทตามความร้อนของเตาอย่างเรียลไทม์จนจบที่ขั้นตอน `DONE`
4. **ความยืดหยุ่นทางเครือข่าย (Offline Support)**: หากทดลองปิดเครื่อง Agent หรือตัดเน็ตตู้ระหว่างขั้นตอนจ่ายเงิน หน้าจอหน้าร้านจะต้องปรับขึ้นแถบข้อความสีน้ำเงิน "กำลังตรวจจับการจ่ายของ" และทำการยิงคำร้องแบบดึงสถานะเบื้องหลัง (Passive Polling) จนกว่า Agent จะกลับมาต่อติดเพื่อดำเนินงานต่อได้อย่างถูกต้อง

## 7. สถานการณ์ B — การลงทะเบียนตู้จำหน่ายใหม่เข้าระบบ

เมื่อต้องการนำตู้จำหน่ายสินค้าเครื่องใหม่เชื่อมเข้าฐานข้อมูลกลาง มีลำดับความปลอดภัย 5 ขั้นดังนี้:

### ขั้นที่ 1 — เปิดสแต็กฐานข้อมูลและ API
```bash
docker compose up -d db server admin-ui
```

### ขั้นที่ 2 — ลงทะเบียนตู้บน Admin Dashboard
1. เปิดเว็บ [http://localhost:3001](http://localhost:3001) และเข้าสู่ระบบหลังบ้าน
2. ไปที่เมนู **Machines** และคลิกปุ่ม **Create / Add Machine**
3. ป้อนรหัสเครื่องที่ต้องการ (เช่น `DEMO-02`) และกำหนดสถานที่ติดตั้ง
4. **⚠️ ข้อสำคัญ:** หน้าจอจะปรากฏรหัส **`secret_token` แบบตัวอักษรธรรมดาเพียงครั้งเดียวเท่านั้น** ให้ทำการคัดลอกรหัสนี้เก็บไว้ทันที (ระบบจะแปลงรหัสเป็น Bcrypt Hash ก่อนจัดเก็บลง MySQL จึงไม่มีโอกาสเรียกดูย้อนหลังได้อีก)

### ขั้นที่ 3 — ตั้งค่าความมั่นคงปลอดภัยบนบอร์ดควบคุม (Agent)
เปิดไฟล์ `.env` ที่ root ของโปรเจกต์ และทำการอัปเดตตัวแปรระบุตนเองของตู้:
```env
MACHINE_CODE=DEMO-02
MACHINE_TOKEN=<วางรหัส secret_token ที่ได้คัดลอกมาจากหลังบ้าน>
```
ทำการสั่งให้ Agent รีสตาร์ทเพื่อโหลดค่าการยืนยันตนใหม่เข้าสู่หน่วยความจำ:
```bash
docker compose up -d --build client
```
*ตรวจจับ Logs ของเซิร์ฟเวอร์ จะต้องพบสัญลักษณ์ `✅ [SocketIO] machine connected: DEMO-02` ผ่านด่านตรวจสอบ*

### ขั้นที่ 4 — กำหนดตัวชี้วัดหน้าตู้ (Machine UI Client)
 Next.js จะฝังตัวแปร `NEXT_PUBLIC_*` ลงไปใน Static Files ตอนที่มีการ Build ดังนั้นเพื่อให้หน้าจอ Kiosk ทราบรหัสความลับรวมถึงชี้ไปยังตู้รหัสใหม่ ให้แก้ไขไฟล์ `.env` :
```env
NEXT_PUBLIC_MACHINE_CODE=DEMO-02
KIOSK_SOCKET_SECRET=<สร้างคีย์ความลับของตู้ใหม่>
NEXT_PUBLIC_KIOSK_SOCKET_SECRET=<ป้อนคีย์ความลับให้ตรงกัน>
```
สั่งบิลด์หน้าจอ UI หน้าร้านใหม่หมดจด:
```bash
docker compose build --no-cache machine-ui
docker compose up -d machine-ui
```

### ขั้นที่ 5 — กำหนดช่องเติมสต็อกสินค้า (Machine Slots)

## 11. สถานการณ์ E — การพัฒนา Backend (Flask)

เหมาะสำหรับนักพัฒนาฝั่ง Backend ที่ต้องการปรับแก้ระบบตรรกะภายในโฟลเดอร์ `server` โดยให้ระบบอัปเดตอัตโนมัติ (Auto-reload) ทันทีที่เซฟไฟล์โดยไม่ต้องนั่งรอบิลด์ Docker Image

### 11.1 รันฐานข้อมูล MySQL รอล่วงหน้าบน Docker
```bash
docker compose up -d db
```

### 11.2 ติดตั้งสภาพแวดล้อมจำลอง (Virtual Environment)
เปิด Terminal และเข้าไปที่ไดเรกทอรี `server/`:
```bash
cd server
python -m venv .venv
```
* **ขั้นตอนเรียกใช้งาน Virtual Env (Windows):**
  ```powershell
  .venv\Scripts\activate
  ```
* **ขั้นตอนเรียกใช้งาน Virtual Env (macOS / Linux):**
  ```bash
  source .venv/bin/activate
  ```

ทำการดาวน์โหลดแพ็กเกจชุดคำสั่งทั้งหมดที่กำหนด:
```bash
pip install -r requirements.txt
```

### 11.3 ตั้งค่าตัวแปรเชื่อมโยง (Local Configuration)
สร้างไฟล์กำหนดตัวแปรของตนเอง ชี้กลับไปยังพอร์ตของ Docker MySQL ที่ได้รับการเผยแพร่ออกมาทางพอร์ต `3307`:

```env
DB_HOST=127.0.0.1

DB_PASSWORD=root
DB_NAME=vending
FLASK_ENV=development
SOCKETIO_ENABLED=1
ALLOW_MOCK_PAY=1
AGENT_BASE_URL=http://localhost:5000
CORS_ORIGINS=http://localhost:3000,http://localhost:3001,http://localhost:3002
```

เปิดการใช้งาน API กลาง:
```bash
python main.py
```
*เซิร์ฟเวอร์จะพร้อมตอบคำสั่งของคุณที่พอร์ต [http://localhost:8000](http://localhost:8000)*

## 15. สถานการณ์ I — การติดตั้ง Agent บน Pi เครื่องจริง

เมื่อต้องการย้ายระบบ Agent ไปรันควบคุมชุดวงจรรวมถึงมอเตอร์จ่ายสินค้าและเครื่องอ่าน NFC จริงบนบอร์ด Raspberry Pi ผ่านสายสัญญาณ LAN

### 15.1 เตรียมการฝั่งเซิร์ฟเวอร์ API ส่วนกลาง
* ตรวจสอบให้มั่นใจว่าเครื่องคอมพิวเตอร์ที่ใช้รันเซิร์ฟเวอร์หลัก (เช่นแล็ปท็อปแอดมินหรือคลาวด์) มีค่า IP ที่บอร์ด Raspberry Pi สามารถเรียกเชื่อมโยงเข้ามาได้ผ่านเครือข่ายภายใน (เช่น `192.168.1.44`)
* ปรับปรุงตัวแปรสภาพแวดล้อม `CORS_ORIGINS` ใน `.env` ของเซิร์ฟเวอร์ให้รองรับ IP ดังกล่าวและสั่งรีสตาร์ท

### 15.2 รันโค้ดฝั่งกล่อง Raspberry Pi จริง
1. ต่อสาย LAN หรือต่อ Wi-Fi วงเดียวกันกับคอมพิวเตอร์เซิร์ฟเวอร์
2. คัดลอกโฟลเดอร์ซอร์สโค้ด `client/agent/` ทั้งหมดไปจัดวางบน Pi
3. สร้างไฟล์กำหนดตัวแปร `.env` ที่โฟลเดอร์นั้นบน Pi:
   ```env
   MACHINE_CODE=MP1-001
   MACHINE_TOKEN=dev-machine-token
   SERVER_SOCKET_URL=http://192.168.1.44:8000
   MACHINE_UI_URL=http://192.168.1.44:3000
   # ตั้งค่าพาธบันทึก SQLite ภายในเครื่อง Pi
   AGENT_DB_PATH=./data/agent.db
   # หากใช้งานบนฮาร์ดแวร์ต่อมอเตอร์และเครื่อง NFC จริง ให้ปรับ sensor mode เป็น gpio
   SENSOR_MODE=gpio
   ```
4. ติดตั้ง Virtual Environment และแพ็กเกจเสริมบนระบบบอร์ด Pi:
   ```bash
   python3 -m venv .venv
   source .venv/bin/activate
   pip install -r requirements.txt
   python agent.py
   ```
   *หลังจากเชื่อมต่อสมบูรณ์ หน้าจอแดชบอร์ดแอดมินจะปรากฏไอคอนสัญญาณออนไลน์สีเขียวของตู้นี้ทันที*

## 16. สถานการณ์ J — การทดสอบบอร์ดควบคุมตรงผ่าน REST API

## 17. สถานการณ์ K — การปรับแต่งพอร์ตหรือการรันบนหลายเครื่อง

หากพบปัญหาว่าเครื่องคอมพิวเตอร์ของคุณมีพอร์ตของบริการตัวอื่นรันทำงานทับซ้อนอยู่ก่อนแล้ว (เช่นมีการรัน MySQL ทับพอร์ต `3307` หรือมีพอร์ต `8000` ถูกใช้งานค้างคาอยู่) ให้ทำตามขั้นตอนการเลี่ยงพอร์ตดังนี้:

1. เปิดแก้ไขไฟล์ [docker-compose.yml](docker-compose.yml)
2. เล็งส่วนของ ports และเปลี่ยนเฉพาะ**ตัวเลขทางด้านซ้ายมือ** (ซึ่งเป็นพอร์ตโฮสต์ฝั่งเครื่องภายนอก) ตัวอย่าง:
   ```yaml
   # แก้ไขจากเดิม "8000:8000" เพื่อขยับ API เลี่ยงไปพอร์ต 8001
   ports:
     - "8001:8000"
   ```
3. เข้าไปปรับแก้ค่าตัวแปรใน `.env` ที่ root ชี้ปลายทาง REST API ให้ตรงตามค่าพอร์ตโฮสต์ล่าสุด:
   ```env
   NEXT_PUBLIC_API_URL=http://localhost:8001
   NEXT_PUBLIC_ADMIN_API_URL=http://localhost:8001
   ```
4. สั่งสั่งทำงานใหม่เพื่อฝังค่าลง Next.js: `docker compose up --build`

## 18. คู่มือการทดสอบชำระเงินผ่าน Omise

โปรเจกต์นี้ติดตั้งสัญญาระบบยืนยันยอดของ Omise เพื่อความสมจริงในการแลกเปลี่ยนค่าเงินโดยใช้ตัวเลือกทดลองจ่ายผ่าน sandbox:

1. **การรูดจ่ายด้วยบัตรเครดิต (Credit Card Payment)**: หน้าร้านหน้าตู้จะมีปุ่มสีเหลืองแสดงตัวอย่าง **`[Test] Simulate Visa Tap`** เมื่อคลิกปุ่มระบบจะส่ง JSON Source ไปยัง backend เพื่อแปลงเป็นบัตรเครดิตชำระเงินโดยอัตโนมัติ
2. **การจ่ายเงินด้วย PromptPay QR**: ระบบจะเจเนอเรตรหัส QR Code ภาพจริงมาขึ้นแสดงบนเครื่องลูกค้า ลูกค้าสามารถคลิกลิงก์สำหรับการจำลองชำระเงิน (Omise Test Tool) หรือยิง webhook จำลองเพื่อปรับสเตตัสธุรกรรมได้
3. **การข้ามขั้นตอนจ่ายเงินด้วย API (Mock Pay Bypass)**: 
   หากไม่ได้ป้อนคีย์จริงของ Omise แต่อยากบังคับออเดอร์ให้เปลี่ยนสถานะเป็นชำระสำเร็จทันทีเพื่อทดสอบมอเตอร์ ให้ยิงสั่งผ่าน POST API ตรงเข้าเซิร์ฟเวอร์ (โดยมีเงื่อนไขต้องตั้งค่าตัวแปร `ALLOW_MOCK_PAY=1` ภายในเครื่อง):
   ```bash
   curl -X POST http://localhost:8000/api/buy/mock-pay \
     -H "Content-Type: application/json" \
     -d '{"charge_id": "chg_test_yourchargeidhere"}'
   ```

## 21. การแก้ปัญหาทั่วไป (Troubleshooting)

### ปัญหาพอร์ตถูกจองใช้งานแล้ว (Port Collision)
* **อาการ:** รันขึ้น Docker แล้วเตือนล้มเหลวที่แถวพอร์ต `3000` หรือ `8000`
* **การแก้ไข:** ปรับพอร์ตทางด้านซ้ายมือใน compose หรือจัดการสั่งปิดแอปที่จองพอร์ตอยู่ เช่น ยุติกระบวนการรันของเครื่องมือเซิร์ฟเวอร์ MySQL บนเครื่องของตัวเอง หรือเลี่ยงไปรัน Scenario K

### การสกัดกั้น Socket ด้วยปัญหา `Authentication failed` ทุกๆ 30 วินาที
* **อาการ:** ใน logs ของ Agent ปรากฏลูปข้อความยืนยันตนล้มเหลวซ้ำๆ ไม่ได้หยุดหย่อน
* **สาเหตุ:** รหัสลับ `MACHINE_TOKEN` ที่ป้อนไม่สอดคล้องกับคีย์ที่เก็บบันทึกบนตาราง MySQL หรือไม่ได้รีสตาร์ทบอร์ดควบคุม Agent หลังมีการแก้ไขตัวแปร Env
* **การแก้ไข:** ล็อกอินเข้าบอร์ด Admin หน้าเว็บเพื่อสร้างตู้จำหน่ายใหม่ และรีบคัดลอกตัวอักษรรหัสธรรมดามาป้อนระบุลง `.env` ให้ตรงกันและสั่งรัน Agent ใหม่

### หน้าจอ Kiosk ขึ้นแสดงข้อความระบุสัญญาณขัดข้องสีแดงเต็มจอ
* **สาเหตุ:** หน้าจอ Next.js ไม่สามารถจับรหัสเกตเวย์ Socket.IO กับตัว Backend Server ได้เนื่องจากคีย์ตรวจสอบ `KIOSK_SOCKET_SECRET` และ `NEXT_PUBLIC_KIOSK_SOCKET_SECRET` ป้อนค่าตัวพิมพ์เล็กพิมพ์ใหญ่คลาดเคลื่อนกัน
* **การแก้ไข:** แก้รหัสความลับให้เหมือนกันในไฟล์ Env และทำตามจังหวะ Scenario F เพื่อ Rebuild หน้าจอลูกค้าตู้ใหม่

### หน้าจอลูกค้าแสดงระบบขัดข้องเตือนตู้กำลังพยายามติดต่อกล่องควบคุม
* **สาเหตุ:** เซิร์ฟเวอร์ API สามารถทำงานได้ปกติ แต่ตัวซอฟต์แวร์จำลองมอเตอร์ควบคุมตู้ (Pi Agent) ใน container หลุดการเชื่อมต่อไป
* **การแก้ไข:** สั่งรีสตาร์ท Agent คอนเทนเนอร์ด้วยคำสั่ง: `docker compose up -d --build client` และเช็คให้แน่ชัดว่าไม่มีขีดข้อความ error ล้มเหลวเกิดขึ้นภายใน logs
